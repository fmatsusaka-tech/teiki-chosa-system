# 開発ロードマップ

## 方針

最初から全機能を作らず、現場で使える最小単位を順番に完成させる。各フェーズは、動作確認と仕様更新を終えてから次へ進む。

## Phase 0: 開発準備

- [x] リポジトリ作成
- [x] プロジェクト憲章
- [x] README
- [x] 初期仕様書
- [ ] 開発技術の選定
- [ ] 開発環境のひな型作成
- [ ] 秘密情報の管理方法を決定
- [ ] テストとCIの基本設定

完了条件：ローカルまたはクラウド開発環境で、空のアプリと自動テストが動く。

## Phase 1: 自由テキスト解析

- 自由形式テキスト入力画面
- 1件・複数件のレコード分割
- 園地・品種・横径・糖度・酸度・備考・計測日の抽出
- 園地名と品種名の正規化
- 警告と未解析文字列の表示
- 解析処理の単体テスト

完了条件：実際の入力例を用いて、複数園地を含む文章を確認画面用データへ変換できる。

## Phase 2: 確認・修正画面

- 複数レコード一覧
- 各項目の編集
- 横径の追加・削除
- 登録対象の切替
- 警告項目の強調
- スマートフォン操作の改善

完了条件：AI解析結果を利用者が修正し、登録対象を確定できる。

## Phase 3: データ登録

- Googleスプレッドシート連携
- 可変長横径の列展開
- 登録日時と計測日の分離
- 重複候補の警告
- 登録成功・失敗の表示
- 再送とエラー処理

完了条件：現行シート互換の形式で、複数レコードを安全に一括登録できる。

## Phase 4: 画像入力

- スクリーンショットアップロード
- 表形式画像から複数レコード抽出
- OCR結果とAI解析の統合
- 元画像を見ながら確認できるUI

完了条件：複数園地が載った実際のスクリーンショットを一括登録できる。

## Phase 5: 写真・音声入力

- 手書きメモ写真の解析
- 音声入力・文字起こし
- 農園固有名称の認識改善
- 現場での入力時間計測と改善

完了条件：スマートフォンだけで、音声または写真から実用的に登録できる。

## Phase 6: 定期調査分析

- 品種・月・調査区分による検索
- 複数園地一覧
- 横径平均、糖度、酸度の比較
- 園地選択式グラフ
- 目標値設定と警告

完了条件：同時期の園地差を一覧とグラフで確認できる。

## Phase 7: 園地分析・気象連携

- 園地別時系列
- 複数年度比較
- 2園地比較
- AMeDAS湯浅の降雨・気温
- 積算温度・積算降雨量
- 表示項目フィルター

完了条件：調査値と気象条件の推移を同じ画面で比較できる。

## Phase 8: 予測・判断支援

- 収穫時横径・糖度・酸度予測
- 異常推移の検知
- 水ストレス管理の参考情報
- 散布・栽培判断の根拠表示
- 予測精度の検証

完了条件：予測の根拠と誤差を示し、人間の判断を補助できる。

## リリース原則

- 各フェーズで実データを使った受入確認を行う
- 仕様変更はドキュメントとIssueへ反映する
- 重要な変更はPull Requestでレビューする
- AIの出力を無確認で本番データへ登録しない
