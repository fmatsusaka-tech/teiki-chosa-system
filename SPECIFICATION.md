# システム仕様書

## 1. 対象

まつさか農園の柑橘定期調査について、入力、確認、保存、閲覧、比較、分析を行うWebアプリケーションを構築する。

本書は初期仕様であり、決定済み事項と未決定事項を分けて管理する。

## 2. 利用者と利用環境

- 主な利用端末はスマートフォン
- 複数のスタッフが入力・閲覧する
- 園地での短時間操作を想定する
- PCでは管理、詳細確認、分析を行う

## 3. 初期リリースの範囲

初期リリースでは次を実現する。

1. 自由形式テキストの入力
2. 1件または複数件の調査レコードへの解析
3. 解析結果の確認・修正
4. 選択したレコードの一括登録
5. Googleスプレッドシートへの出力

画像、音声、分析機能は後続段階で追加する。

## 4. 調査レコード

初期項目は以下とする。

| 項目 | 必須 | 型 | 備考 |
|---|---:|---|---|
| recordId | 自動 | 文字列 | システム内の一意ID |
| measuredAt | 必須 | 日付 | 計測日。入力にない場合は登録日を初期値とする |
| orchard | 必須 | 文字列 | 園地の正規名称 |
| variety | 必須 | 文字列 | 品種の正規名称 |
| diameters | 任意 | 数値配列 | 横径。件数は可変 |
| brix | 任意 | 数値 | 糖度 |
| acidity | 任意 | 数値 | 酸度 |
| remarks | 任意 | 文字列 | 備考 |
| registeredAt | 自動 | 日時 | 実際に登録した時刻 |
| sourceType | 自動 | 列挙 | text、voice、screenshot、photo、pdf |
| sourceText | 自動 | 文字列 | 解析前の原文 |
| warnings | 自動 | 配列 | 解析上の警告 |

調査項目は将来追加できるよう、画面、検証、出力処理を密結合させない。

## 5. 入力仕様

### 5.1 自由形式

利用者へ固定書式を要求しない。

例：

```text
徳田、早生、39.6-40.5-42.7-40.0-32.9、糖度8.4、酸度3.8
```

```text
7月15日
徳田 早生 39.6 40.5 42.7 40.0 32.9 糖8.4 酸3.8
寺門 極早生 41.2 40.8 39.9 糖7.9 酸3.2 少し小玉
```

区切り文字は、読点、カンマ、空白、改行、ハイフン等を許容する。

### 5.2 複数レコード

1回の入力に複数園地が含まれる場合、園地単位のレコードへ分割する。

解析結果は配列で返す。

### 5.3 計測日

- 入力内に日付があれば採用する
- 複数レコード共通の日付表記は各レコードへ適用する
- 日付がない場合は登録時点の日付を仮設定する
- 仮設定した場合は確認画面で分かるようにする

### 5.4 数値

- 全角数字を半角へ正規化する
- 単位記号や「糖」「糖度」「酸」「酸度」を識別する
- 横径と糖酸を数値の位置だけで断定しない
- 不確実な場合は空欄または警告とし、勝手に確定しない

## 6. 名称の正規化

### 6.1 品種

入力表記を正規名称へ変換する。

例：

| 入力例 | 正規名称 |
|---|---|
| 早生 | 早生（宮川・興津など） |
| ごくわせ、極わせ | 極早生 |

正規化辞書はコードへ直接埋め込まず、設定として管理する。

### 6.2 園地

表記揺れ、ひらがな、呼称を正規名称へ変換する。

例：

| 入力例 | 正規名称 |
|---|---|
| とくだ、徳田園、Tokuda | 徳田 |

園地マスターは既存データから作成し、管理画面または設定ファイルで変更可能にする。

## 7. AI解析結果

解析APIは少なくとも次を返す。

```json
{
  "records": [],
  "globalWarnings": [],
  "unparsedText": ""
}
```

各レコードは、抽出値に加えて次を保持する。

- 項目別の確信度
- 警告
- 原文上の根拠範囲（実装可能な場合）

確信度の低い項目はUIで強調する。

## 8. 確認・修正画面

登録前に必ず確認画面を表示する。

利用者は各レコードについて次を行える。

- 園地の変更
- 品種の変更
- 計測日の変更
- 横径の追加、修正、削除
- 糖度、酸度、備考の修正
- レコードの登録対象／対象外切替
- レコード削除

複数レコードは1画面でまとめて確認し、一括登録できる。

## 9. バリデーション

最低限、次を検証する。

- 園地と品種が選択されている
- 数値項目が数値として解釈できる
- 明らかに不自然な値には警告を出す
- 同一内容の重複登録の可能性を警告する

許容範囲や目標値は設定可能にし、固定値にしない。

## 10. Googleスプレッドシート出力

内部では `diameters` を配列として保持する。

出力時に次のように展開する。

```text
タイムスタンプ / 計測日 / 園地 / 品種 / 横径1 / 横径2 / ... / 糖度 / 酸度 / 備考
```

- 測定個数より多い横径列は空欄
- 当初は既存シートとの互換性を優先する
- 将来20個以上へ拡張できること
- 内部データ構造とシート列構造は分離する

## 11. 分析機能

### 11.1 定期調査分析

検索条件：

- 品種
- 月
- 調査区分

調査区分の初期定義：

- 前半：前月25日から当月10日
- 中頃：当月11日から20日

表示項目：

1. 園地名
2. 備考
3. 横径平均
4. 糖度
5. 酸度
6. 収穫時横径予測
7. 収穫時糖度予測
8. 収穫時酸度予測

表示園地を選択でき、グラフでは色とマーカーで識別する。

### 11.2 園地分析

検索条件：

- 園地
- 品種
- 必要に応じて備考または処理区

機能：

- 単園地の時系列
- 複数年度比較
- 2園地比較
- 横径、糖度、酸度の表示切替
- AMeDAS湯浅の降雨、最高気温、最低気温との重ね合わせ
- 積算温度、積算降雨量

## 12. 将来機能

- スクリーンショットから複数レコード抽出
- 手書きメモ写真の解析
- 音声入力最適化
- PDF取り込み
- 果実数、果実重、LWP、葉色、果皮色
- 目標値からの逸脱検知
- 収穫時予測
- 水ストレス管理支援
- 散布・栽培判断支援

## 13. 非機能要件

- スマートフォン優先のレスポンシブUI
- 通信が不安定な現場でも入力内容を失いにくいこと
- 一覧・グラフ表示が実用的な速度であること
- 個人情報や認証情報をリポジトリへ保存しないこと
- APIキー等は環境変数で管理すること
- 操作ログとエラーログを確認できること
- データのバックアップおよび再出力が可能であること
