# 入力解析ルール

音声に関する記述は、スマートフォン標準の音声入力で作成された文章を解析する場合を指す。本システム独自の音声認識・文字起こしは実装しない。

## 目的

利用者へ固定書式を強制せず、現場で自然に入力された文章を調査レコードへ変換する。

## 解析順序

1. 文字の正規化
2. 日付など全体共通情報の抽出
3. レコード境界の推定
4. 園地・品種の候補抽出
5. 横径、糖度、酸度の抽出
6. 名称の正規化
7. バリデーション
8. 警告と未解析文字列の生成

## 文字の正規化

- 全角数字・記号を半角へ変換
- 連続空白を整理
- 改行はレコード境界候補として保持
- 読点、カンマ、タブ、スラッシュを区切り候補として扱う
- 小数点は保持する
- ハイフンは負数ではなく、横径の区切りとして使われる場合がある

## 日付

次のような表記を許容する。

- 2026/7/15
- 2026-07-15
- 7月15日
- 7/15
- 今日、昨日（音声入力では利用時点の日本時間を基準に解決）

年が省略された場合は、登録時点に最も自然な年を候補とする。ただし年境界付近など曖昧な場合は警告する。

## レコード分割

以下を組み合わせて境界を推定する。

- 改行
- 園地マスターに一致する語の出現
- 品種名の出現
- 表形式の行構造
- 同じ項目パターンの繰り返し

1行に複数園地が含まれる場合も、園地候補を基準に分割する。

## 横径

- 横径は複数の数値列として抽出する
- 個数は固定しない
- 糖度・酸度としてラベル付けされた数値は横径から除外する
- 単位がmmと明示された場合は横径候補を強める
- 不自然な範囲の値は削除せず警告する

## 糖度と酸度

優先するラベル：

- 糖度、糖、Brix、ブリックス
- 酸度、酸、クエン酸

数値のみの場合は、並び順や過去形式を参考に候補を作るが、確信が持てなければ警告する。

`%` は入力として許容するが、内部値は数値のみとする。

## 備考

主要項目へ分類できなかった文章のうち、意味のある説明を備考候補とする。

未解析文字列と備考は区別し、解析に失敗した断片を自動で備考へ混ぜない。

## 名称の正規化

正規化はマスターの以下を使う。

- 正式名称
- 別名
- ひらがな・カタカナ表記
- 音声認識で起こりやすい誤変換

複数候補が同程度の場合は勝手に決定せず、候補選択を求める。

## 警告例

- 計測日がなかったため登録日を仮設定
- 園地名に複数候補あり
- 品種名を正規化
- 糖度と酸度の判別が不確実
- 横径に通常範囲外の値あり
- 未解析文字列あり
- 同じ計測日・園地・品種の登録済みデータあり

## 解析例

入力：

```text
7/15 徳田 早生 39.6-40.5-42.7-40.0-32.9 8.4 3.8 少し小玉
```

期待結果：

```json
{
  "measuredDate": "2026-07-15",
  "orchardName": "徳田",
  "varietyName": "早生（宮川・興津など）",
  "diameters": [39.6, 40.5, 42.7, 40.0, 32.9],
  "brix": 8.4,
  "acidity": 3.8,
  "remarks": "少し小玉"
}
```

数値ラベルがないため、糖度・酸度の解釈には確信度または警告を付ける。
