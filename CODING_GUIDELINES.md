# コーディングガイドライン

## 1. 基本原則

- 現場の入力速度と安全性を優先する
- 読みやすく、変更しやすいコードを書く
- 調査項目、園地名、品種名、しきい値をハードコードしない
- AI、Google Sheets、OCR等の外部サービスは交換可能な境界を設ける
- 不明な仕様を推測で固定せず、IssueまたはTODOへ残す

## 2. 言語と命名

- コード上の変数・関数・ファイル名は原則英語
- 画面表示と利用者向けメッセージは日本語
- 略語を乱用しない
- 真偽値は `is`、`has`、`can` 等で始める
- 日付と日時は名称で区別する（例：`measuredDate`、`registeredAt`）

## 3. 構造

少なくとも次の責務を分離する。

- UI
- 入力の前処理
- AI解析
- 名称正規化
- バリデーション
- データ保存
- Googleスプレッドシート出力
- 分析・集計

外部サービス固有の処理を画面コンポーネントへ直接書かない。

## 4. データ

- 横径は内部では数値配列として扱う
- 元の入力文字列を保持する
- 計測日と登録日時を別々に保持する
- 欠損値と数値0を区別する
- 単位を暗黙に混在させない
- 日時は内部でタイムゾーンを明示し、表示時に日本時間へ変換する

## 5. AI解析

- AIの応答は構造化データとして検証する
- 応答を信用して直接保存しない
- 必須フィールド、型、範囲をアプリ側で再検証する
- 確信度の低い値や矛盾を警告として返す
- プロンプトと解析モデルをコードから分離し、変更履歴を残す
- 同じ入力を再現できるテストケースを整備する

## 6. エラー処理

- 利用者が次に何をすべきか分かる日本語メッセージを表示する
- 外部API失敗時に入力内容を失わない
- 一括登録では、全件成功・一部成功・全件失敗を区別する
- ログへ秘密情報や個人情報を出力しない

## 7. テスト

最低限、次をテストする。

- 1件の標準入力
- 複数園地の入力
- 横径個数が異なる入力
- 日付なし・日付あり
- 園地名と品種名の表記揺れ
- 全角数字、改行、読点、ハイフン
- 糖度または酸度が欠ける入力
- 解釈不能な入力
- 外部API失敗
- Google Sheets出力列の整合性

不具合修正時は、再発防止テストを追加する。

## 8. セキュリティ

- APIキー、認証情報、スプレッドシートの秘密情報をコミットしない
- `.env` はGit管理対象外とする
- `.env.example` には項目名のみ記載する
- 入力値を検証し、画面表示時は適切にエスケープする
- 権限は必要最小限にする

## 9. Git運用

- 1つのIssueまたは目的につき、変更範囲を小さくする
- コミットメッセージは変更内容が分かる命令形または簡潔な説明にする
- 大きな変更はブランチとPull Requestを使用する
- Pull Requestには目的、変更点、確認方法、未解決事項を書く
- 仕様変更を伴う場合は同じPRでドキュメントも更新する

## 10. 完了の定義

機能は次をすべて満たして完了とする。

- 要件を満たす
- 自動テストが通る
- スマートフォンで主要操作を確認した
- エラー時の挙動を確認した
- 必要なドキュメントを更新した
- 秘密情報が含まれていない
