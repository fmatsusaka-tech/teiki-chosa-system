# CONTRIBUTING.md

この文書は、本リポジトリで人間またはAIが変更を行う際の標準手順です。

## 1. 基本フロー

```text
要望・不具合
  ↓
GitHub Issue
  ↓
設計確認
  ↓
作業ブランチ
  ↓
実装・テスト
  ↓
Pull Request
  ↓
レビュー
  ↓
Squash and merge
  ↓
ブランチ削除
```

`main`へ直接コミットしないでください。

## 2. Issue

変更前にIssueを作成します。

Issueには最低限、次の内容を含めます。

- 背景と目的
- 対象範囲
- 非対象
- 完了条件
- 必要なテスト
- 関連する既存Issueや文書

1 Issue = 1 PRを基本とします。大きな機能は、レビュー可能な単位へ分割してください。

## 3. ブランチ

推奨形式は次のとおりです。

```text
feature/<short-name>
fix/<short-name>
docs/<short-name>
refactor/<short-name>
test/<short-name>
```

ブランチ名は短く、変更目的が分かる英語表現にします。

## 4. 実装前の確認

- 現在の`main`を基準にする
- 対象コードと関連テストを読む
- 既存の命名規則と責務分離に従う
- Issue範囲外の変更を混ぜない
- 秘密情報や個人情報を含めない

AIエージェントは、リポジトリ直下の`AGENTS.md`にも従ってください。

## 5. コミット

作業中のコミットは分割して構いませんが、最終的なマージは原則`Squash and merge`を使用します。

コミットメッセージには、変更内容が分かる表現を使用してください。

## 6. Pull Request

### タイトル

```text
feat: 機能名 (#Issue番号)
fix: 修正内容 (#Issue番号)
docs: 文書内容 (#Issue番号)
refactor: 変更内容 (#Issue番号)
test: テスト内容 (#Issue番号)
```

### 本文

PR本文には次を記載します。

```markdown
## 関連Issue
- Closes #123

## 目的

## 変更内容

## 設計判断

## テスト
- `npm test`
- `npm run typecheck`

## 未実装事項

## 後続課題
```

コード変更時は、実行したテストコマンドと結果を必ず記載してください。

## 7. レビュー

レビューでは次を確認します。

- Issueの目的を満たしているか
- Issue範囲外の変更がないか
- 欠測値と0を混同していないか
- AI/OCR固有実装が上位層へ漏れていないか
- 既存用語とデータ構造を維持しているか
- エラー時にアプリ全体が停止しないか
- テストが十分か
- 秘密情報が含まれていないか

指摘事項が解決するまでマージしません。

## 8. マージ

- 全Checks成功を確認する
- コンフリクトがないことを確認する
- 原則`Squash and merge`を使用する
- Squash後のタイトルにIssue番号を残す
- マージ後に作業ブランチを削除する
- 完了したIssueが閉じたことを確認する

## 9. 文書更新

仕様、環境変数、操作方法、設計方針が変わる場合は、コードだけでなく関連文書も更新してください。

設計思想に関する判断は`PROJECT_PRINCIPLES.md`、AI作業ルールは`AGENTS.md`、人間向け手順は本ファイルへ反映します。
